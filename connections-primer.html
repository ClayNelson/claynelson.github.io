Why team size compounds

A two-minute explanation for why adding one developer makes coordination harder than it looks — and why AI agents make it dramatically worse.

The basic idea: Every person on a team needs to stay in sync with every other person. As you add people, the number of those sync relationships grows much faster than headcount. This isn’t a technology problem — it’s a fundamental property of teams.

Step 3 — five developers, each running 3 AI agents

Now each developer isn’t working alone — they’re running 3 AI agents in parallel. Each agent is making changes to the codebase. That’s 15 active workers, not 5.

105 potential collision pairs — none of which know about each other

The agents belonging to Dev 1 don’t know what the agents belonging to Dev 2 are doing. They all have access to the same codebase. When two of them make changes to the same file, no one is warned in advance — the problem only appears when the code is merged.

The full picture across team sizes:

The key insight: Developer pairs grow manageable with structure. Agent pairs grow faster and are completely invisible — no agent knows what any other agent is doing, unless there’s a platform coordinating them. That platform is GitHub. The pull request is what forces agents to negotiate before merging.

This is why the fight isn’t about which AI model is smarter. A smarter agent working in isolation still creates the same invisible collision problem. The question is whether the agents are operating on shared ground — shared codebase state, shared decisions, shared memory of why things were built the way they were.
